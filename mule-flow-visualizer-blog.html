Internal Tooling / MuleSoft

Stop reading raw XML to understand your own Mule flows.

Mule Flow Visualizer turns a folder of MuleSoft XML into an interactive diagram, a request tracer, an error-coverage map, and a branch-level cost calculator — all in a single HTML file that runs entirely in your browser.

01
The problem

Flow logic lives in XML. Understanding lives in someone's head.

Any MuleSoft engineer who has inherited an integration knows the routine: open the XML, mentally stack the <choice> blocks, trace a <flow-ref> into another file, lose your place, and start again. The logic is all there — it's just not readable at a glance.

Built for the moment you're handed someone else's project

Mule Flow Visualizer was built to answer four questions fast, without spinning up Anypoint Studio or exporting anything to a diagramming tool:

  • What does this flow actually do, end to end?
  • Which flow would handle this specific incoming request?
  • If this error type fires, does anything actually catch it?
  • Which branch is expensive, which one is dead code, and what would it cost at real traffic volumes?
02
Core capability

Interactive flow diagrams, straight from your XML

Load one or many .xml files and every <flow> and <sub-flow> is parsed into a tree the tool can render, search, and navigate.

A readable diagram for every flow

Choice routers, scatter-gather, try scopes, foreach, async, and error handlers all render as their own distinct node shapes, so the branching structure is visible instead of buried in indentation.

Sections collapse and expand individually, or all at once, so a 400-line flow can be skimmed at the top level and drilled into only where it matters.

Flow-ref navigation

Click a flow-ref node and jump straight into the referenced flow, wherever it lives across your loaded files.

Dependency graph

A separate map view shows which flows call which, and lets you find every caller of a given flow instantly.

In-diagram search

Search across the rendered diagram with next/previous match jumping, auto-expanding any collapsed parents.

SVG export

Export the current diagram as a clean SVG — for documentation, a design review, or a wiki page.

03
Core capability

Paste a request, get the exact execution path

Instead of guessing which flow handles an incoming call, paste the raw request and let the tool match it for you.

cURL or raw HTTP in, matched flow out

Paste a curl command or a raw HTTP request block. It's parsed for method, path, headers, and body, then scored against every HTTP listener path across your loaded flows to find the best match — including path parameters like /orders/{id}.

  • Step-through debugger: once matched, walk the flow processor by processor with Next / Prev / Stop controls.
  • Variable inspector: see Mule variables as they're set, updated, or removed at each step of the trace.
  • Basic expression evaluation: simple #[...] DataWeave-style expressions are evaluated against the request context so you can see resolved values, not just the raw expression.
04
Core capability

Know your error coverage before production does

Error handling in Mule is scoped and inherited in ways that are easy to get subtly wrong. This view makes the gaps visible.

Every processor, mapped to its error handler

The tool walks every processor in every loaded flow and resolves which error-handler chain actually wraps it — including handlers inherited from a parent flow through a flow-ref call.

Trace a specific error type. Query something like HTTP:CONNECTIVITY or MULE:EXPRESSION and the tool finds the exact catch block that would handle it — or tells you plainly that nothing does — plus a likely cause and fix for common error types.
05
Core capability

The Branch Calculator: complexity, dead code, and cost

This is the analysis view built for the conversation after "does this flow work" — the one about maintainability and spend.

Cyclomatic complexity, computed automatically

Every choice and scatter-gather block in a flow is located and used to compute cyclomatic complexity (M), giving a concrete maintainability signal instead of a gut feeling. Branches that can never be reached — a catch-all placed before a real condition, for example — are flagged as dead.

06
Under the hood

One HTML file. No install, no server, no data leaving your machine

🔒

Everything — XML parsing, diagram rendering, request tracing, cost math — runs client-side in plain JavaScript. Your flow XML, and whatever request payloads you paste in, never leave the browser tab. That makes it safe to point at client or production integration code without a second thought about where the file goes.
